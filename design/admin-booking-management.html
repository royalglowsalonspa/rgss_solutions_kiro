🔀 States, Flows & Error Handling

Collapsible sections covering every booking state and edge case from the admin perspective.

⏳ Pending State — Awaiting Admin Action

Entry: Customer completes booking wizard → status = PENDING

Admin UI:

  • Booking appears in /admin/bookings list with amber badge
  • NEW badge shown if booking is < 5 minutes old
  • ⚠️ STALE indicator if pending > 2 hours (QStash check)
  • Ably real-time: row appears instantly without refresh
  • Toast notification with booking summary

Admin Actions:

  • Assign staff — Dropdown per service (filtered by availability + skills)
  • Approve — Transitions to CONFIRMED, triggers SMS + email
  • Reject — Opens rejection modal (reason required)

Customer Actions (while pending):

  • Can cancel → removes from admin list (Ably event)
  • Can edit services/time → admin list updates (Ably event)
// Staff assignment validation
if (!allServicesHaveStaff) {
  showWarning("Assign staff to all services before approving");
  return; // Soft block — can override
}
✅ Confirmed State — Customer Checked In

Entry: Admin approves booking → status = CONFIRMED

Side Effects on Confirmation:

  • SMS sent to customer with booking details + salon address
  • Confirmation email with calendar attachment (.ics)
  • Staff notified via Ably (staff:[staffId] channel)
  • Customer booking card updates to green "Confirmed"

Admin Actions from Confirmed:

  • Mark In Progress — Customer arrives, starts services
  • Reschedule — Change date/time (customer notified)
  • Cancel (salon-initiated) — Triggers apology email + priority rebooking
  • Mark No-Show — After appointment end time passes

Auto No-Show Alert:

  • QStash job fires 15 minutes after booking end time
  • If status still CONFIRMED → alert admin with "Mark as No-Show?" prompt
  • Admin confirms manually (no auto-transition)
// QStash no-show check (runs 15min after booking.end_time)

  "check_time": "2025-01-15T12:45:00"
}

💳 Checkout Flow — Invoice Generation

Entry: Admin clicks "Checkout" on IN_PROGRESS booking

Step 1 — Apply Offer:

  • Dropdown shows eligible offers for this customer
  • Filtered by: membership tier, first-visit status, service category
  • Discount preview shown immediately
  • Only one offer can be applied per booking

Step 2 — Payment Method:

  • Options: Cash, Card (POS), UPI, Split Payment
  • Split payment opens sub-form for amount allocation
  • No online payment gateway — all in-salon

Step 3 — Review & Generate Invoice:

  • Invoice preview: services table, subtotal, GST 18% (SAC code 999721), total
  • "Complete & Generate Invoice" button
  • On success: PDF generated, emailed to customer, gems awarded

Post-Checkout Side Effects:

  • Invoice PDF generated (stored in S3)
  • Email with PDF attachment sent to customer
  • Gems awarded: 1 gem per ₹100 spent (floor)
  • Meta CAPI: Purchase event fired with revenue value
  • Brevo: customer profile updated (last_visit, total_spent, gems_balance)
// Invoice calculation
const subtotal = services.reduce((sum, s) => sum + s.price, 0);
const discount = offer ? calculateDiscount(offer, subtotal) : 0;
const taxable = subtotal - discount;
const gst = Math.round(taxable * 0.18); // 18% GST
const total = taxable + gst;
const gemsEarned = Math.floor(total / 100); // 1 gem per ₹100
🚫 No-Show Flow — Tracking & Escalation

Trigger: Manual admin action or auto-alert after 15min past appointment end

No-Show Marking Process:

  • Admin clicks "Mark No-Show" on confirmed booking
  • Confirmation dialog: "Are you sure? This affects the customer's record."
  • On confirm: status → NO_SHOW, counter incremented

No-Show Tier System:

CountActionCustomer Impact
1st no-show Warning email sent Informational only
2nd no-show SMS + email warning Visible warning on profile
3rd no-show Final warning Flag on admin dashboard
4th no-show booking_requires_approval = true All future bookings need manager approval

Escalation at 4 No-Shows:

  • Customer's booking_requires_approval flag set to true
  • All future bookings from this customer route to manager for approval
  • Regular receptionist cannot approve these bookings
  • Manager/owner can reset the flag manually
// No-show counter logic
const noShowCount = customer.no_show_count + 1;
await db.customer.update({
  where: { id: customer.id },
  data: {
    no_show_count: noShowCount,
    booking_requires_approval: noShowCount >= 4
  }
});
🚶 Walk-in Booking Creation

Purpose: Quick booking for customers who arrive without appointment

Walk-in Flow:

  • Admin clicks "New Walk-in" button on dashboard
  • Search existing customer by phone or create new profile
  • Select services + assign available staff
  • Booking created directly as CONFIRMED (skips PENDING)
  • Can immediately mark as IN_PROGRESS

Walk-in Rules:

  • Status starts at CONFIRMED (no approval needed)
  • No no-show tracking for walk-ins (they're already present)
  • booking.source = "walk_in" flag set
  • Same checkout flow applies (invoice, gems, etc.)
  • Customer gets SMS with booking confirmation

❌ Rejection Modal — Reason Required

Trigger: Admin clicks "Reject" on pending booking

Modal Contents:

  • Booking summary (customer name, date, services)
  • Reason dropdown: "Staff unavailable", "Slot conflict", "Service unavailable", "Other"
  • Free-text notes field (required if "Other" selected)
  • "Confirm Rejection" button (red, destructive action)

On Rejection:

  • Status → REJECTED with reason stored
  • Customer sees reason in their booking history
  • Email notification with reason + "Rebook" CTA link
  • SMS: "Your booking at Royal Glow was not confirmed. Reason: [reason]"

🏪 Salon-initiated Cancellation

Trigger: Admin cancels a confirmed/pending booking on behalf of salon

Scenarios:

  • Staff emergency/leave announced after booking confirmed
  • Equipment failure or salon closure
  • Double-booking discovered

Cancellation Flow:

  • Admin selects reason from dropdown
  • Optional: suggest alternative date/time
  • "Cancel Booking" confirmation (shows customer impact)

Customer Notification:

  • Apology email with reason and rebooking link
  • SMS notification immediately
  • Priority rebooking: customer flagged for next available slot
  • No negative impact on customer record (no no-show count)

// Invoice data model
interface Invoice {
  id: string;                    // UUID
  booking_id: string;            // FK → bookings
  invoice_number: string;        // "INV-2025-0042"
  customer_id: string;           // FK → customers
  salon_id: string;              // FK → salons

  // Line items
  items: {
    service_id: string;
    service_name: string;
    staff_id: string;
    staff_name: string;
    base_price: number;          // In paise (₹500 = 50000)
    discount_amount: number;
    taxable_amount: number;
  }[];

  // Totals
  subtotal: number;              // Sum of base_price
  discount_total: number;        // Offer/coupon discount
  taxable_amount: number;        // subtotal - discount
  gst_rate: 18;                  // Fixed 18%
  gst_amount: number;            // taxable * 0.18
  total: number;                 // taxable + gst
  sac_code: "999721";            // Beauty & grooming SAC

  // Payment
  payment_method: "cash" | "card" | "upi" | "split";
  payment_split?: {
    method: string;
    amount: number;
  }[];

  // Offer applied
  offer_id?: string;
  offer_code?: string;
  offer_discount_type: "percentage" | "fixed";
  offer_discount_value: number;

  // Loyalty
  gems_earned: number;           // floor(total / 100)
  gems_redeemed: number;         // If customer used gems

  // Metadata
  generated_at: DateTime;
  pdf_url: string;               // S3 signed URL
  emailed_at?: DateTime;
  created_by: string;            // Admin user ID
}

👥 Staff Assignment Logic

// Staff assignment for booking services
async function getAvailableStaff(serviceId: string, bookingDate: Date, timeSlot: string) {
  // 1. Get staff who can perform this service
  const qualifiedStaff = await db.staffService.findMany({
    where: { service_id: serviceId, is_active: true }
  });

  // 2. Filter by schedule (not on leave)
  const onDutyStaff = qualifiedStaff.filter(s =>
    !isOnLeave(s.staff_id, bookingDate) &&
    isScheduled(s.staff_id, bookingDate, timeSlot)
  );

  // 3. Filter by existing bookings (not double-booked)
  const availableStaff = onDutyStaff.filter(s =>
    !hasConflictingBooking(s.staff_id, bookingDate, timeSlot)
  );

  // 4. Sort by workload (least busy first)
  return availableStaff.sort((a, b) =>
    getTodayBookingCount(a.staff_id) - getTodayBookingCount(b.staff_id)
  );
}

// Assignment rules:
// - Admin can override and assign any qualified staff
// - Unassigned services show warning (not blocking)
// - Staff gets notified via Ably on assignment
// - Re-assignment allowed until IN_PROGRESS status
