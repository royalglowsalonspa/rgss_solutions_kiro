🔑 Authentication Strategy

Royal Glow uses Google OAuth only — no email/password authentication. This simplifies the UX, eliminates password-reset flows, and leverages Google's security infrastructure.

AspectDetail
ProviderGoogle OAuth 2.0 (via NextAuth.js)
Scopesopenid profile email
SessionJWT stored in httpOnly cookie
Token RefreshAutomatic via NextAuth rotation
Calendar AccessIncremental consent — asked AFTER first booking confirmed
No-Password PolicyNo email/password forms, no magic links, no OTP

🚀 Flow Overview

The auth flow is triggered when an unauthenticated user performs a protected action (e.g., "Book Now"). Context is stored in sessionStorage before redirecting.

  1. User triggers an auth-requiring action (Book Now, Join Membership, etc.)
  2. App stores context in sessionStorage['rgss_auth_context']
  3. Redirect to /sign-in
  4. User clicks "Sign in with Google" → Google OAuth consent screen
  5. OAuth callback received → session created
  6. First-time user? → Redirect to /onboarding
  7. Returning user? → Redirect to / + restore context
  8. Onboarding: collect phone, DOB, gender, consent → save profile
  9. Redirect to / → read sessionStorage → restore action
  10. Context restored: if book=1 → open booking dialog; if leadId → link lead

⚡ Key Design Decisions

  • sessionStorage over localStorage — context dies with tab, no stale data across sessions
  • Google Calendar incremental consent — asked only after first booking is confirmed, not at sign-in
  • Cookie consent integration — if user consents on onboarding, banner is suppressed for those categories
  • Session expiry re-auth — user re-authenticates but skips onboarding (profile already complete)
  • No password reset flow — eliminated entirely by Google-only strategy
  • Lead linking — if user arrived via lead capture, leadId in context links anonymous lead to authenticated user

// Session check middleware
export function middleware(request: NextRequest) {
  const session = await getToken({ req: request });
  if (!session) {
    // Store current page context for post-auth restoration
    const url = new URL('/sign-in', request.url);
    url.searchParams.set('callbackUrl', request.nextUrl.pathname);
    return NextResponse.redirect(url);
  }
}

Key point: Re-auth after session expiry skips onboarding entirely. The user's profile is already complete in the database. Only a fresh JWT is needed.

⚡ Onboarding Side Effects (6 Steps)

When POST /api/onboarding/complete succeeds, these 6 side effects fire in order:

  1. Save Profile — Write user profile to database (name, phone, DOB, gender, email). Mark onboardingComplete: true.
  2. Assign Source — Store utm_source and utm_campaign on user record. If leadId present, link anonymous lead → authenticated user.
  3. Write Consent to localStorage — Store consent preferences in localStorage['rgss_cookie_consent'] so cookie banner knows what's already approved.
  4. Sync to Brevo — Create/update contact in Brevo (Sendinblue) with email, phone, name. Add to "Customers" list. Tag with source.
  5. Fire Meta CAPI — Server-side CompleteRegistration event to Meta Conversions API. Include fbp/fbc cookies if available.
  6. Send Welcome Email — Queue welcome email via Brevo transactional API. Template includes first booking CTA and salon info.
// Side effects implementation (server-side)
export async function completeOnboarding(userId: string, data: OnboardingData) {
  // 1. Save profile
  await db.user.update({ where: { id: userId }, data: { ...profile, onboardingComplete: true } });

  // 2. Assign source + link lead
  if (data.leadId) await db.lead.update({ where: { id: data.leadId }, data: { userId } });
  await db.userSource.create({ data: { userId, source: data.utm_source, campaign: data.utm_campaign } });

  // 3. Consent written client-side (returned in response for client to store)

  // 4. Sync Brevo
  await brevo.createContact({ email: user.email, attributes: { FIRSTNAME: data.name, PHONE: data.phone } });

  // 5. Fire Meta CAPI
  await metaCapi.track('CompleteRegistration', { userId, source: data.utm_source });

  // 6. Send welcome email
  await brevo.sendTransactional({ templateId: WELCOME_TEMPLATE, to: user.email, params: { name: data.name } });
}

🍪 Cookie Consent localStorage Structure

Cookie consent preferences stored in localStorage['rgss_cookie_consent']. If user consented on onboarding, the banner is suppressed for those categories.

// localStorage key: 'rgss_cookie_consent'
interface CookieConsent {
  version: number;         // Schema version (currently 1)
  timestamp: string;       // ISO date when consent was given
  source: 'onboarding' | 'banner'; // Where consent was collected
  categories: {
    essential: true;       // Always true, cannot be changed
    analytics: boolean;    // Google Analytics, Vercel Analytics
    marketing: boolean;    // Meta Pixel, Google Ads
  };
}

// Example: user consented to analytics on onboarding
localStorage.setItem('rgss_cookie_consent', JSON.stringify({
  version: 1,
  timestamp: '2025-05-28T10:30:00.000Z',
  source: 'onboarding',
  categories: {
    essential: true,
    analytics: true,
    marketing: false
  }
}));

Banner Suppression Logic

// Cookie banner component
function CookieBanner() {
  const consent = JSON.parse(localStorage.getItem('rgss_cookie_consent'));

  // If consent exists (from onboarding or previous banner interaction)
  // → suppress banner entirely
  if (consent?.categories) return null;

  // Otherwise show banner
  return <BannerUI />;
}

🔄 Session Expiry Handling

When a session expires, the user re-authenticates but does NOT go through onboarding again.

ScenarioBehaviour
JWT expires (30 days)NextAuth attempts silent refresh. If refresh fails → 401.
API returns 401Client stores context in sessionStorage → redirects to /sign-in
User clicks "Sign in with Google"OAuth flow completes → session restored
Post-auth checkonboardingComplete: true → skip /onboarding
Context restorationRead sessionStorage → restore previous action
Google Calendar scopeIf previously granted, token is refreshed automatically
 
// Session expiry detection + context preservation
function useAuthGuard() {
  const { data: session, status } = useSession();

  useEffect(() => {
    if (status === 'unauthenticated') {
      // Store current context before redirecting
      const context: AuthContext = {
        returnTo: window.location.pathname,
        timestamp: Date.now(),
      };
      sessionStorage.setItem('rgss_auth_context', JSON.stringify(context));

      // Show toast
      toast.info('Session expired. Please sign in again.');

      // Redirect after brief delay (let toast show)
      setTimeout(() => router.push('/sign-in'), 1500);
    }
  }, [status]);
}

📅 Google Calendar — Incremental Consent

Google Calendar access is NOT requested at sign-in. It uses incremental consent — asked only after the first booking is confirmed.

WhenWhat Happens
First sign-inScopes: openid profile email only
First booking confirmedPrompt: "Add to Google Calendar?" → triggers incremental consent
User grants calendarScope added: calendar.events → booking events auto-sync
User declinesNo calendar sync, no further prompts for 30 days
Subsequent bookingsIf scope already granted → auto-add event (no prompt)
 
// Incremental consent for Google Calendar
async function requestCalendarAccess() {
  const result = await signIn('google', {
    callbackUrl: window.location.href,
    scope: 'https://www.googleapis.com/auth/calendar.events',
    prompt: 'consent', // Force consent screen for new scope
  });
  return result;
}
